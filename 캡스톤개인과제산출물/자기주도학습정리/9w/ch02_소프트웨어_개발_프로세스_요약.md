# Chapter 02 · 소프트웨어 개발 프로세스

> **학습 목표** : 다양한 SW 프로세스 모델의 구조·장단점을 비교하고, 실무 맥락에서 적합한 모델을 선택할 수 있다.

---

## 1. 프로세스 & SDLC 기본 개념

| 개념 | 설명 |
|------|------|
| **프로세스** | 목적 달성을 위해 순서가 정해진 일련의 절차 |
| **프로세스의 목적** | 축적된 노하우 전달 → 시행착오 감소 → guide 역할 |
| **프로세스 3요소** | 인원(능력·동기 부여) / 프로세스(절차·방법) / 도구·장비 |
| **SDLC** | SW를 어떻게 개발할 것인가에 대한 전체 흐름을 체계화한 모델. 계획 수립 ~ 폐기까지 순차 단계로 구성 |

**SW 프로세스 모델의 목적**
- 공장 생산 방식처럼 SW 개발 전 과정을 하나의 프로세스로 정의
- 예산·자원 범위 내에서 **고품질** SW 생산

## 2. 프로세스 모델 전체 맵

```
주먹구구식  →  선형(폭포수)  →  V 모델
                                     ↓
           진화적(프로토타입) ← 한계 극복
                                     ↓
             나선형 / 단계적 개발 / 통합(UP) / 애자일
```

---

## 3. 주먹구구식 모델 (Build & Fix)

**특징** : 공식 가이드라인·설계 단계 없이 코드 작성 후 수정 반복

```
코딩(구현) → 제품 → 문제? ──아니오──→ 사용
                  ↑    │예
                  └────수정
```

| 적합한 경우 | 단점 |
|------------|------|
| 1인 개발, 단기 프로젝트 (학기 수업 과제 수준) | 문서·산출물 없음 → 유지보수 불가 |
| | 아키텍처 설계 불가, 진척 파악 불가 |
| | 누적 수정으로 코드 구조 붕괴 |

---

## 4. 선형 순차적 모델 (Waterfall)

**별칭** : Linear Sequential / Waterfall / Classic Life Cycle

**단계 흐름**
```
계획 → 요구분석 → 설계 → 구현 → 테스트 → 유지보수
 (각 단계마다 검증(Verification) 수행)
```

| 장점 | 단점 |
|------|------|
| 관리 용이, 체계적 문서화 | 앞 단계 완료 전 다음 단계 불가 |
| 요구변화가 적은 프로젝트에 적합 | 각 단계 결과물의 완벽성 요구 |
| | 중간 가시적 결과 없음 → 고객 답답함 |

> **핵심 키워드** : 순차(Sequential), 단방향(No feedback loop), 산출물(Artifact) 중심

---

## 5. V 모델

**특징** : 폭포수 + **테스트 단계 명시적 확장** → 각 개발 단계를 검증하는 데 초점

```
요구분석 ←─────────────────── 시스템 테스트 ─→ 인수 테스트
   ↓          요구분석 검증                           ↑
아키텍처 설계 ←──────────── 통합 테스트
   ↓          인터페이스 검증           ↑
모듈 설계 ←─────── 단위 테스트
   ↓      모듈 검증       ↑
          구현
```

> 폭포수(산출물 중심) vs V 모델(**검증(Verification & Validation) 중심**)

## 6. 진화적 프로세스 모델 (Prototype 계열)

### 6-1. 프로토타입이란?
- 완성 전에 사용자 요구를 확인하기 위한 **시제품(모형)**
- 사용자와 의사소통 도구로 활용

### 6-2. 실험적 프로토타입 모델
```
요구분석 → 빠른설계 → 프로토타입 개발 → 고객평가
                 ↑           │ 아니오
                 └── 수정요구사항 ◄──┘
                              │ 예
                    최종사용자 요구 → 분석 → 설계 → 구현
```
> 프로토타입은 **버린다(throwaway)**. 최종 요구 확정 후 정식 개발 시작.

### 6-3. 진화적 프로토타입 모델
```
요구분석 → 빠른설계 → 프로토타입 개발 → 평가?
                 ↑           │ 아니오           │ 예
                 └── 수정요구사항 ◄──┘    최종 제품
```
> 프로토타입을 **그대로 발전**시켜 최종 제품화.

**프로토타입 개발 3단계**
1. 요구사항 정의·분석 (n차 반복으로 정제)
2. 빠른 설계 (UI 중심, 사용자 인터페이스 집중)
3. 프로토타입 개발 (완제품 ×, 입출력 확인 목적)

## 8. 통합 프로세스 모델 (UP / Unified Process)

**등장 배경** : 폭포수 모델의 경직성 → **반복적 개발 방법**으로 해결

### 반복적 개발 구조
```
R₁→A₁→D₁→C₁→T₁→M₁
               ↓
   R₂→A₂→D₂→C₂→T₂→M₂
               ↓
   R₃→A₃→D₃→C₃→T₃→M₃
```
`R=요구정의, A=분석, D=설계, C=구현, T=테스트, M=유지보수`

### UP 4단계 (가로축 = 단계, 세로축 = 작업 영역)

| 단계 | 영어 | 핵심 활동 |
|------|------|----------|
| **도입** | Inception | 비즈니스 모델링·요구사항 정의 집중 |
| **구체화** | Elaboration | 분석·설계 집중, 아키텍처 확정 |
| **구축** | Construction | 구현·테스트 집중, 반복(n회) |
| **전이** | Transition | 배치(Deployment), 사용자 이전 |

**공통 작업** (전 단계에 걸쳐 지속) : 형상(변화) 관리 / 프로젝트 관리 / 환경 점검

### 비즈니스 모델링
- 시스템이 운영될 비즈니스 환경 이해·분석
- 도메인 모델(Domain Model) 구축 → 이후 설계 단계의 클래스 구조로 연결
- **BPMN** : 업무 흐름을 시각화하는 글로벌 표기법 (수영 레인 + 플로우 차트)

## 9. 애자일 프로세스 모델

### 애자일 선언문 4대 가치

| 기존 중심 | → | 애자일 중심 |
|----------|---|------------|
| 프로세스·도구 | | **개개인·상호 소통** |
| 문서 | | **실행 가능한 SW** |
| 계약·협상 | | **고객과의 협력** |
| 계획 | | **변화에 대한 민첩한 대응** |

### 개발 방법
```
프로토타입 개발 → 사용자 확인 → 일부 기능 사용  (반복)
```
반복적 개발 + 잦은 출시(Frequent Release) 목표

### 스크럼 (Scrum)

> "럭비 스크럼처럼 똘똘 뭉쳐 짧은 구간을 **Sprint**한다"

**특징**
- 팀 개선·프로젝트 관리에 초점 (≠ 구체적 개발 절차 명시)
- 경험적 관리 기법 (Empirical process control)

**스크럼 진행 5단계**
1. 제품 기능 목록 작성 (Product Backlog)
2. 스프린트 계획 회의 → 스프린트 구현 목록 선정
3. 스프린트 수행 (1~4주) + 일일 스크럼 회의
4. 스프린트 개발 완료 → 최종 제품 증분 생성
5. 스프린트 완료 후 → 검토 회의 + 회고(Retrospective)

**주요 산출물** : 소멸 차트(Burndown Chart), 스프린트 구현 목록

## 10. 모델 비교 요약표

| 모델 | 반복성 | 요구변경 대응 | 문서화 | 적합 규모 |
|------|--------|-------------|--------|----------|
| 주먹구구식 | ✗ | 낮음 | ✗ | 초소규모 |
| 폭포수 | ✗ | 낮음 | ◎ | 중~대규모, 요구 고정 |
| V 모델 | ✗ | 낮음 | ◎ | 신뢰성 중시 시스템 |
| 프로토타입 | △ | 높음 | △ | 요구 불명확 프로젝트 |
| 나선형 | ○ | 높음 | ○ | 고위험 대규모 |
| 통합(UP) | ◎ | 높음 | ○ | 중~대규모 |
| 애자일/스크럼 | ◎ | 매우 높음 | △ | 변화 잦은 중소규모 |

## 11. 핵심 용어 정리

| 용어 | 정의 |
|------|------|
| SDLC | Software Development Life Cycle |
| Waterfall | 단방향 순차 개발, 폭포수 |
| Prototype | 사용자 요구 확인용 시제품 |
| UP | Unified Process, 통합 프로세스 |
| Sprint | 스크럼의 짧은 반복 개발 주기 (1~4주) |
| Backlog | 개발해야 할 기능 목록 |
| BPMN | Business Process Model and Notation |
| Burndown Chart | 잔여 작업량 추적 소멸 차트 |

## 12. 자주 나오는 시험 포인트 ⚡

- **폭포수 vs V 모델** : 산출물 중심 vs 검증 단계 명시
- **실험적 vs 진화적 프로토타입** : 프로토타입을 버리는가 vs 발전시키는가
- **UP 4단계 순서** : 도입 → 구체화 → 구축 → 전이
- **애자일 선언문 4가지 가치** 암기
- **스크럼 ≠ 구체적 개발 방법론** : 팀 운영 지침(경험적 관리 기법)
- **주먹구구식 단점** : 문서 없음, 아키텍처 부재, 구조 붕괴

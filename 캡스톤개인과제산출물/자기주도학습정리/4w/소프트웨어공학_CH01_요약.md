## Section 01. 소프트웨어의 이해

### 1. 소프트웨어란?

단순한 프로그램(source code)을 넘어서는 포괄적 개념이다.

- 원시 코드(source code)
- 자료구조, DB 구조, 테스트 결과 등 **모든 산출물**
- 각 개발 단계에서 생산된 **문서**
- **사용자 매뉴얼**

> ✅ 핵심: SW = 프로그램 + 그 이상의 산출물 전체

---

### 2. 소프트웨어 vs 하드웨어 특성 비교

| 구분 | H/W | S/W |
|------|-----|-----|
| 생산 방식 | **제조** (정해진 틀, 인력 간 차이 적음) | **개발** (개인 능력 차이 매우 큼) |
| 노후화 방식 | **소모** (부품 마모, 고장 빈도 증가) | **품질 저하** (닳지 않음, 기능 동일 유지) |

---

### 3. 실패 곡선 비교

#### (a) H/W 실패 곡선 — 욕조 곡선(Bathtub Curve)

```
실패율
  ↑
  |*        *
  | *      *
  |  ******
  +--------→ 시간
  초기높음 → 안정 → 마모 후 다시 증가
```

- 초기 실패율 높음 → 오류 해결 → 안정 운용 → 마모로 인해 다시 실패율 상승

#### (b) 이상적인 S/W 실패 곡선

- 초기 미발견 오류로 실패율 높음 → 오류 해결 후 **폐기 시까지 낮은 실패율 유지**
- 전제 조건: 개발 완료 후 변경 사항 없음, 환경 변화 없음

#### (c) 실제 S/W 실패 곡선

- **기능 추가/수정(변경)** 발생 시 부작용으로 **실패율이 급격히 스파이크** 형태로 상승
- 변경이 반복될수록 이상적 곡선과 격차 증가 → 점진적 품질 저하(degradation)

---

### 4. 소프트웨어 위기(Software Crisis)

- S/W 발전 속도가 H/W 발전 속도를 따라가지 못함
- H/W는 **검증된 부품 조립** 방식, S/W는 **처음부터 개발**하는 방식 → 근본적 차이
- 해결 방안: **CBD(Component-Based Development) 개발 방법론** 도입

---

### 5. 소프트웨어 개발의 어려움

| 규모 비유 | 특성 |
|-----------|------|
| 개집 짓기 | 혼자 가능, 설계 도면 불필요, 단순 |
| 단독주택 짓기 | 설계사·도면 필요, 다수 인원 참여 |
| 대형 빌딩 짓기 | 조직·통제·조정 필요, 고려 사항 다수 |

**대규모 S/W 개발의 3대 어려움:**

1. **개발 과정이 복잡하다** → 복잡성 감소 방법론 제시
2. **참여 인력이 많다** → 팀 구성 및 효율적 관리 방안 필요
3. **개발 기간이 길다** → PMBOK 기반 프로젝트 관리 필요

## Section 02. 공학과 소프트웨어 공학의 이해

### 1. 공학(Engineering)의 특성

- 과학적 지식을 활용해 문제를 해결하되, **한정된 기간과 비용**이라는 제약 조건 하에서 수행

### 2. 소프트웨어 공학의 정의

> 품질 좋은 소프트웨어를 **경제적으로** 개발하기 위해  
> 계획·개발·유지·관리의 전 과정에서  
> **공학·과학·수학적 원리와 방법**을 적용하여  
> 필요한 이론, 기술, 도구를 연구하는 학문

### 3. 소프트웨어 공학의 목적

- S/W 개발의 어려움 해결
- 효율적 개발을 통한 **생산성 향상**
- **고품질** 소프트웨어 제품 생산

### 4. SDLC (Software Development Life Cycle)

계획 → 분석 → 설계 → 구현 → 테스트 → 유지보수

## Section 03. 소프트웨어 개발 단계 소개

### 1. 소프트웨어 개발 프로세스 6단계

| 단계 | 내용 | 주요 도구/개념 |
|------|------|----------------|
| **1. 계획** | 비용 산정, 일정 계획, 위험 관리 | COCOMO, FP 모델, WBS, CPM |
| **2. 요구분석** | 문제 파악 → 요구사항 도출 → 다이어그램 작성 | DFD, E-R 다이어그램, UML 유스케이스 |
| **3. 설계** | 아키텍처 설계, 모듈 설계 | GoF 디자인 패턴, 응집도/결합도 |
| **4. 구현** | 코딩 | 표준 코딩 규칙 |
| **5. 테스트** | 검증 및 확인 | 단계별·목적별·품질특성별 분류 |
| **6. 유지보수** | 릴리즈 후 관리 | 수정/적응/기능보강/예방 유지보수 |

---

### 2. 주요 도구 정리

#### WBS (Work Breakdown Structure)
- 프로젝트 목표 달성을 위한 전체 업무를 **최하위 작업 패키지 단위로 계층화**한 구조도
- 예: 냄비 밥 짓기 → 착수 / 계획 / 실행 / 감시·통제 / 종결

#### 소프트웨어 개발 프로세스 모델 (2장에서 상세 다룸)

- 주먹구구식 모델
- **폭포수(Waterfall) 모델** (선형순차적)
- V 모델
- 프로토타입 모델 (진화적)
- **나선형(Spiral) 모델**
- 단계적 개발 모델
- 통합 프로세스 모델 (UP)
- **애자일(Agile) 프로세스 모델**
